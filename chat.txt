This is a critical moment in your development. Firestore Security Rules are the only thing preventing a student from "marking" their own work or a parent from seeing another child's grades.

Since you are using Claude Opus or a similar agent, you should give it a very specific ruleset. Here is the logic we are implementing to ensure maximum security on a budget:

The Multi-Tenant Logic
Global Lock: Everything is allow read, write: if false; by default.

Role Verification: We check a users collection to see if the person logged in is a teacher, student, or parent.

School Isolation: Every check ensures request.auth.token.schoolId == resource.data.schoolId.

The Firestore Rules Code
Copy this into your firestore.rules file or give it to your AI agent as the "Source of Truth" for security:

JavaScript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    
    // Helper: Is the user logged in?
    function isSignedIn() {
      return request.auth != null;
    }

    // Helper: Get user data from the 'users' collection
    function getUserData() {
      return get(/databases/$(database)/documents/users/$(request.auth.uid)).data;
    }

    // --- RULES FOR TEACHERS ---
    match /students/{studentId} {
      // Teachers can see students in their own school
      allow read: if isSignedIn() && getUserData().role == 'teacher' 
                  && getUserData().schoolId == resource.data.schoolId;
    }

    match /marks/{markId} {
      // Teachers can create and edit marks for their school
      allow write: if isSignedIn() && getUserData().role == 'teacher'
                   && getUserData().schoolId == request.resource.data.schoolId;
    }

    // --- RULES FOR PARENTS ---
    match /marks/{markId} {
      // Parents can ONLY read marks if the mark belongs to their child
      // This assumes the mark document contains a 'studentId' field
      allow read: if isSignedIn() && getUserData().role == 'parent'
                  && resource.data.studentId in getUserData().childrenIds;
    }

    // --- RULES FOR STUDENTS ---
    match /assignments/{assignmentId} {
      // Students can read assignments for their class/school
      allow read: if isSignedIn() && (getUserData().role == 'student' || getUserData().role == 'teacher')
                  && getUserData().schoolId == resource.data.schoolId;
    }
  }
}
How this works for your Parents
When the school admin creates a student, your backend logic (or a Cloud Function) should update the Parent's document in the users collection to include an array called childrenIds.

Safety: Even if a parent figures out the "ID" of another student's marks, Firebase will block the request because that student's ID isn't in their childrenIds list.

Cost-Saving Tip (Performance)
In the rules above, I used a get() function to check the user's role. Every time a user opens a page, this "Rule Check" counts as 1 Document Read in Firebase.

To make it 100% free: Tell your AI agent to use "Custom Claims" for the role and schoolId. Custom claims are stored inside the user's login token, so checking them in Security Rules is free and doesn't count as a database read.
